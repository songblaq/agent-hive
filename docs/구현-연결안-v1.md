# AgentHive 구현 연결안 v1

작성일: 2026-03-09
상태: 현재 코드 기준 정렬 메모

## 1. 현재 판단

AgentHive는 더 이상 빈 스캐폴드 단계가 아니다.
이미 아래가 존재한다.

- CLI 명령 진입점 (`src/cli/index.ts`)
- 명령 구현 파일 (`init`, `project`, `task`, `status`, `setup`)
- 허브 초기화 로직 (`src/core/hub.ts`)
- 프로젝트 등록 로직 (`src/core/registry.ts`)
- task lifecycle 일부 (`src/core/task-manager.ts`)
- BACKLOG 재생성 (`src/core/backlog.ts`)
- slug 유틸 (`src/utils/slug.ts`)
- YAML 모델/검증 계층 (`src/models/*`)
- YAML 템플릿 + Markdown 템플릿 (`templates/*`)

즉, 현재 목표는 “무에서 만들기”가 아니라,
**이미 구현된 조각들을 MVP 일관성에 맞게 연결/정리/수정하는 것**이다.

## 2. 현재 확인된 정렬 상태

### 2.1 잘 되어 있는 부분

- `agenthive init` → 허브 생성 흐름 존재
- `agenthive project add` → slug 생성 + registry 반영 + 프로젝트 디렉터리 생성 존재
- `agenthive task create/claim/complete/list` → 기본 흐름 존재
- `agenthive status` → kanban 출력 존재
- `agenthive setup` → 포인터 파일 생성 흐름 존재
- hub와 repo의 분리 구조는 코드/문서 양쪽에서 일관적이다.

### 2.2 아직 더 맞춰야 하는 부분

- 템플릿 파일이 존재하지만 일부 코드는 여전히 inline 문자열로 파일을 생성한다.
- summary/plan/message/review/backlog 템플릿이 CLI와 직접 연결되지 않았다.
- review 상태/owner 해제/lock 해제 규칙이 아직 운영 규약과 완전히 일치하는지 추가 확인이 필요하다.
- project/task 생성 시 출력 문구와 실제 상태 변화의 표준화가 더 필요하다.

## 3. 우선 정렬 대상

### 3.1 TASK-003 init

현재:
- 허브 디렉터리와 기본 파일을 생성한다.

다음 정렬 포인트:
- `templates/config.yaml`, `templates/registry.yaml` 활용 여부 결정
- `PROTOCOL.md`를 내장 상수로 둘지, 템플릿/문서 파일에서 읽을지 결정
- display language prompt 정책을 유지하되 non-interactive 옵션을 명확히 한다.

추천:
- config/registry는 모델 기반 생성 유지 가능
- `PROTOCOL.md`는 추후 템플릿 분리 검토

### 3.2 TASK-004 project add

현재:
- registry 업데이트
- project hub 구조 생성
- `project.yaml` 작성
- `BACKLOG.md` 생성

다음 정렬 포인트:
- `templates/project.yaml` 사용 여부 결정
- context/ 기본 파일 생성까지 포함할지 결정
- duplicate registration 메시지 표준화

추천:
- 최소한 `project.yaml`과 `BACKLOG.md`는 템플릿 기반 또는 템플릿 구조와 동일하게 맞춘다.

### 3.3 TASK-005 task lifecycle

현재:
- task 디렉터리 생성
- `task.yaml`, `plan.md`, `summary.md` 생성
- claim 시 lock 생성 + doing 전환
- complete 시 lock 제거 + done 전환
- BACKLOG 재생성

다음 정렬 포인트:
- `templates/task.yaml`, `templates/plan.md`, `templates/summary.md`, `templates/message.md`, `templates/review.md` 소비 방식 확정
- createTask가 inline 문자열 대신 template source를 사용하도록 정리
- `complete`에서 review/done 전환 시 lock 처리 정책 명확화
- scope overlap warning 메시지 형식 정리

추천:
- TASK-005가 template consumption의 첫 수혜자가 되어야 한다.

## 5. 템플릿 소비 전략 제안

### 옵션 A: 전부 파일 템플릿 기반
장점:
- 사람이 편집 가능
- 오픈소스 템플릿 자산으로 명확
- 도구 간 공유 쉬움

단점:
- 치환 로직 필요
- 템플릿/모델 불일치 위험 증가

### 옵션 B: YAML은 코드 생성, Markdown은 파일 템플릿
장점:
- 구조적 데이터는 타입 안정성 유지
- 사람이 자주 보는 문서는 템플릿 재사용 가능
- 현재 구조에서 가장 마찰이 적음

단점:
- 생성 경로가 두 가지라 약간 섞여 보일 수 있음

### 추천
**MVP는 옵션 B가 가장 적절하다.**

- YAML: 코드에서 타입 안전하게 생성
- Markdown: `templates/*.md` 기반 생성

이 방식이 현재 코드와 가장 잘 맞고,
다른 에이전트가 동시에 작업해도 충돌이 적다.

## 6. 운영 메모

현재 BlaqKlawBot의 역할은 코드 구현 경쟁이 아니라,
아래를 우선 보강하는 것이 효율적이다.

- 문서 정렬
- 템플릿 계층 보강
- 결정 기록
- 개밥먹기 운영 해석 고정
- 다른 에이전트 결과 수렴

즉, 구현 코어는 Claude Code / Codex가 밀고,
나는 그 위에서 **운영 정합성 레이어**를 계속 강화하는 식이 적절하다.
