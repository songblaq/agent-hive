# AgentHive 분리 전략 v1

작성일: 2026-03-09
상태: 확정 v2

## 1. 왜 분리하는가

AgentHive 본체는 프로토콜/CLI/허브 상태 관리 도구다.
하지만 사용자가 확인 가능한 가시적 결과물은 별도의 대시보드가 더 적합하다.

다만 이걸 repo까지 지금 분리하면 공유 schema와 운영 문맥이 너무 빨리 갈라진다.

따라서 현재 결론은 아래와 같다.

- 제품 표면은 둘로 분리한다.
  - `agent-hive`
    - 역할: 프로토콜, CLI, 허브 상태 관리, task lifecycle
  - `agent-hive-dashboard`
    - 역할: 허브 상태를 사람이 확인할 수 있는 웹 UI
- 그러나 **repo는 지금 하나로 유지한다.**
  - dashboard는 같은 repo 안 `apps/dashboard`로 시작한다.

## 2. 역할 경계

### agent-hive
- hub 초기화
- 프로젝트 등록
- task 생성/claim/complete
- pointer file 생성
- 상태 파일 읽기/쓰기
- source of truth 관리

### agent-hive-dashboard
- `~/.agenthive/` 읽기 전용 또는 안전한 조회 계층
- 프로젝트 목록 표시
- kanban/task 상태 표시
- 최근 로그/결정/리뷰 표시
- 향후: live refresh / 필터 / drill-down

## 3. 왜 repo까지는 지금 분리하지 않는가

초기 MVP에서 repo를 분리하면 아래 비용이 바로 생긴다.

- shared model/schema 중복
- hub reader 로직 중복
- version/release 관리 복잡도 증가
- 개밥먹기 문맥 분산
- 여러 에이전트가 동시에 두 repo를 오가야 하는 운영 비용 증가

지금은 CLI와 dashboard가 같은 허브 파일 구조를 공유하므로,
하나의 repo 안에서 product surface만 나누는 편이 더 현실적이다.

권장 구조:

```text
agent-hive/
  src/                  # CLI/core
  templates/            # hub/task templates
  apps/
    dashboard/          # user-visible web app
```

## 4. 사용자 확인 가능 목표

사용자가 최종적으로 확인해야 하는 것은 "사이트"다.
즉, AgentHive Dashboard는 최소한 아래를 보여줘야 한다.

- 등록된 프로젝트 목록
- 각 프로젝트 task 현황
- doing/review/blocked 상태 강조
- 최근 activity log
- 주요 decision/retrospective 링크
- "지금 무엇이 진행 중인지"를 한 화면에서 볼 수 있는 landing view

## 5. 권장 구현 단계

### Phase 1
- 정적/로컬 웹 UI
- 허브 파일 읽기
- 프로젝트 목록 + kanban board 표시
- 로컬에서 바로 열어볼 수 있는 단일 dashboard 페이지

### Phase 2
- task 상세 보기
- 로그/리뷰/decision 보기
- 새로고침 UX 개선

### Phase 3
- watcher 또는 polling
- 실시간 상태 반영
- Symphony/OpenClaw 연결

## 6. repo 분리 재검토 조건

아래 조건이 생기면 repo 분리를 다시 검토한다.

- dashboard 배포 cadence가 CLI와 완전히 달라질 때
- dashboard가 별도 backend/service를 갖게 될 때
- 외부 공개 제품으로 branding/release가 분리될 때
- shared schema가 안정화되어 package 경계가 명확해질 때

## 7. 현재 결론

앞으로는 AgentHive를 “엔진”, AgentHive Dashboard를 “관측 창”으로 본다.

정확한 실행 구조는 다음이다.

- repo: `agent-hive` 하나 유지
- 제품 표면: `agent-hive` + `agent-hive-dashboard`
- dashboard 위치: `apps/dashboard`

이 방향이면 사용자는 실제 결과물을 눈으로 확인할 수 있고,
에이전트 협업 시스템도 단순 대화형 프로토콜을 넘어 제품화 방향을 갖게 된다.
