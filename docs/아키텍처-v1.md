# AgentHive 아키텍처 v1

작성일: 2026-03-09
상태: 초안 v1

## 1. 목적

AgentHive는 서로 다른 AI 도구(Claude Code, Codex, OpenCode, Cursor, Copilot, ChatGPT 등)가
같은 프로젝트를 파일 기반으로 협업할 수 있게 하는 프로토콜이자, 이후 CLI와 자동화 런타임으로
확장될 수 있는 운영 레이어다.

핵심 목표는 다음과 같다.

- 서로 다른 에이전트가 같은 규칙으로 협업한다.
- 상태 저장은 중앙 허브에서 일관되게 관리한다.
- 프로젝트 repo에는 최소 포인터와 규칙만 둔다.
- 초기에는 수동/반자동으로 시작하고, 이후 CLI/자동화로 확장한다.
- 첫 번째 개밥먹기 대상은 AgentHive 자기 자신이다.

## 2. 3계층 구조

### 2.1 공개 프로젝트 레이어

경로 예시:
- `~/_/projects/agent-hive`

역할:
- 오픈소스 repo
- CLI 소스
- 템플릿
- 문서
- 설치 스크립트
- 예제

원칙:
- 공개 가능한 것만 둔다.
- 운영 상태, 개인 로그, 실제 작업 락 데이터는 두지 않는다.

### 2.2 개인 허브 레이어

경로 예시:
- `~/.agenthive/`

역할:
- 실제 협업 상태 저장소
- 프로젝트 등록 정보
- task 카드
- lock
- decision 기록
- append-only 로그
- 에이전트 프로필

원칙:
- source of truth는 여기다.
- 여러 repo가 같은 허브를 참조한다.
- 개인/private 데이터는 repo가 아니라 hub에 둔다.

### 2.3 프로젝트 포인터 레이어

위치:
- 각 실제 프로젝트 repo 내부

예시 파일:
- `AGENTS.md`
- `CLAUDE.md`
- `.agenthive.pointer.yaml`
- `.cursor/rules/...`
- `.github/copilot-instructions.md`

역할:
- 이 프로젝트가 AgentHive를 사용한다는 사실을 선언
- 허브 위치 및 프로젝트 slug를 연결
- 각 도구별 최소 규칙 전달

## 3. Symphony와의 관계

AgentHive와 Symphony는 경쟁 관계가 아니라 계층 분리 관계다.

### 3.1 Symphony의 역할

- 모든 요청의 단일 intake
- 상위 관제 채널
- 프로젝트 선택과 우선순위 조정
- 어떤 에이전트 조합으로 회의/실행할지 결정
- 상태 요약 보고
- 사람이 보는 control plane

### 3.2 AgentHive의 역할

- 실제 작업 상태 저장
- task lifecycle 관리
- lock, handoff, review, decision 기록
- append-only 협업 기록 유지
- machine-readable state plane

### 3.3 운영 원칙

- Discord 채널은 진실 원장이 아니다.
- 실제 상태는 `~/.agenthive/`에 저장한다.
- Symphony는 intake/control plane이다.
- AgentHive는 state plane이다.

## 4. 핵심 운영 원칙

### 4.1 One Task, One Owner, One Scope

- 하나의 task에는 한 시점에 한 owner만 존재한다.
- task는 수정 대상 파일 범위를 명시한다.
- scope가 겹치는 활성 task는 자동 또는 수동 점검이 필요하다.

### 4.2 Plan Before Modify

- 구현 전 `plan.md`가 있어야 한다.
- 구현은 승인된 계획 범위 안에서만 진행한다.

### 4.3 Review After Modify

- 구현 후 builder와 다른 에이전트가 review를 담당한다.
- 최소 1회 이상 검토 기록이 남아야 한다.

### 4.4 Append-Only 우선

- messages, reviews, log는 append-only를 기본으로 한다.
- 타 에이전트의 기록은 수정하지 않는다.
- `summary.md`만 현재 owner가 최신 상태 스냅샷으로 갱신한다.

## 5. 초기 workflow 모델

### 5.1 Conference

용도:
- 설계 토론
- 비교 검토
- 옵션 평가

패턴:
- 같은 질문을 여러 에이전트에게 던짐
- 결과를 비교
- 다수결 또는 점수화로 의사결정

### 5.2 Pipeline

용도:
- 실제 구현
- 리뷰
- 수정 반영

패턴:
- Planner → Builder → Reviewer → Arbiter

### 5.3 Kanban

용도:
- 병렬 작업 분배
- 작업 보드 기반 운영

패턴:
- backlog / ready / doing / review / done

## 6. MVP 범위

v1 MVP는 아래만 우선 구현한다.

- hub 초기화
- 프로젝트 등록
- task 생성
- task claim/lock
- backlog/status 조회
- 포인터 파일 생성

제외:
- 자동 할당
- watcher daemon
- 웹 대시보드
- 고급 분산 실행
- 외부 이슈 트래커 연동

## 7. 첫 번째 개밥먹기 대상

첫 번째 프로젝트는 `agent-hive` 자기 자신이다.

이유:
- 프로토콜을 실제로 검증할 수 있다.
- 규칙의 불편함이 즉시 드러난다.
- 다른 대형 프로젝트에 넣기 전에 다듬을 수 있다.

## 9. 리스크

- scope 충돌 판정 규칙이 아직 약하다.
- lock lease 만료 정책이 미정이다.
- Symphony와 AgentHive의 자동화 경계가 아직 추상적이다.
- 여러 에이전트가 동시에 구조를 바꾸면 초반 충돌 가능성이 높다.

## 10. 현재 결론

AgentHive v1은 지금 바로 시작 가능한 수준의 프로토콜이다.
다만 제품이 아니라 운영 규약에 더 가깝기 때문에,
초기 구현은 복잡한 자동화보다 파일 기반 + CLI 중심으로 단순하게 시작하는 것이 맞다.
