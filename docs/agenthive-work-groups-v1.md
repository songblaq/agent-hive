# AgentHive Work Groups v1

작성일: 2026-03-10
상태: 운영 설계 초안

## 1. 핵심 구분

AgentHive에서는 작업을 크게 두 그룹으로 나눈다.

### A. Project Level Up 그룹
목적:
- 프로젝트 자체의 성숙도를 올린다.
- 직접 업무를 수행하기보다, 프로젝트를 검토/분석하고 필요한 업무를 만든다.

대표 역할:
- 버그/품질 검토
- 보안 점검
- 디자인/UX 점검
- 기획 보강 포인트 탐색
- 구조 리서치
- 후속 task 생성
- decision / retrospective 정리

즉, 이 그룹은 **업무를 만드는 그룹**이다.

### B. 업무 수행 그룹
목적:
- 이미 정의된 task를 실제로 수행한다.
- 구현, 수정, 리뷰, 정리, 산출물 생성을 담당한다.

대표 역할:
- 코드 구현
- 리팩터링
- 테스트 작성
- 리뷰 반영
- 문서 정리
- UI/기능 구현

즉, 이 그룹은 **업무를 하는 그룹**이다.

## 2. 중간층 — Skill / Role Design 그룹

두 그룹 사이에는 중간층이 필요하다.

목적:
- 어떤 업무에 어떤 스킬이 필요한지 정의
- 어떤 에이전트가 어떤 역할에 적합한지 정리
- planner / builder / reviewer / researcher / designer 같은 역할 고도화
- Kira나 다른 에이전트가 사용할 스킬 팩 정리

이 그룹은 아래를 다룬다.
- 역할 정의
- 스킬 요구사항
- 에이전트 배치 전략
- skill pack 고도화
- 어떤 모델/에이전트가 어떤 일에 강한지 기록

## 3. OpenClaw와의 관계

- OpenClaw는 이 세 그룹을 오케스트레이팅한다.
- AgentHive는 그룹별 task/plan/review/log를 저장한다.
- Dashboard는 현재 상태를 사람에게 보여준다.

## 4. 주기 운영 방식

### Project Level Up 크론
- 프로젝트를 주기 점검
- 개선 과제 발굴
- 후속 task 생성 또는 제안

### Skill / Role Design 크론
- 스킬 부족 영역 탐지
- 역할 배치/에이전트 적합성 재검토
- 반복 작업의 스킬화 후보 수집

### Execution 그룹
- 주기적으로 점검받되, 실제 수행은 task 단위로 위임

## 5. 왜 이 구조가 필요한가

이 구조가 없으면 다음 문제가 생긴다.
- 눈앞의 업무만 처리하고 프로젝트 품질 성숙이 느려짐
- 누가 어떤 역할에 적합한지 계속 즉흥적으로 정함
- 반복 작업이 스킬/역할 자산으로 축적되지 않음

따라서 AgentHive는 단순 task board가 아니라,
**프로젝트 성숙도 + 역할 설계 + 실제 수행**을 함께 다루는 구조로 가야 한다.
