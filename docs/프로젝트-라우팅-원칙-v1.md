# AgentHive 프로젝트 라우팅 원칙 v1

작성일: 2026-03-09
상태: 초안

## 1. 목적

사용자가 어느 repo 경로에서 요청하든, AgentHive가 현재 작업 경로를 기준으로 올바른 프로젝트를 찾아 해당 hub 프로젝트에 연결되도록 한다.

## 2. 기본 원칙

- 기본 project selection은 현재 작업 경로(`cwd`) 기준이다.
- 현재 경로가 registry에 등록된 프로젝트 경로와 일치하거나 그 하위 경로면 해당 프로젝트로 resolve한다.
- macOS symlink 경로 차이(`/var` vs `/private/var`)는 canonical path 비교로 흡수한다.
- 명시적 slug 지정이 있으면 그것을 우선할 수 있다.
- 여러 후보가 있으면 가장 긴 경로(prefix가 가장 구체적인 항목)를 우선한다.

## 3. 사용자 경험 관점

- 사용자는 "아무 repo에서나 요청해도 현재 프로젝트에 맞게 하이브가 일한다"고 느껴야 한다.
- 매번 프로젝트를 수동 선택하게 만드는 UX는 최소화한다.
- 잘못된 프로젝트로 resolve될 가능성이 있으면 경고하되, 기본값은 현재 path 기준 자동 선택이다.

### 우선순위 규칙
1. 사용자가 명시한 `project slug`
2. 현재 작업 경로 `cwd`와 registry canonical path 매칭
3. 더 긴 경로 우선 매칭
4. 어느 것도 없을 때만 수동 선택 또는 첫 프로젝트 fallback

## 4. Dashboard와의 관계

- 실행 라우팅은 현재 path 기준이다.
- 관측용 dashboard는 허브 전체 프로젝트 목록을 보여준다.
- 즉, 실행은 현재 프로젝트 중심이고, 시야는 다중 프로젝트 전체를 본다.

### dashboard 선택 규칙
- 프로젝트 목록은 항상 전체 허브 기준으로 렌더링한다.
- 기본 선택은 현재 path와 매칭되는 프로젝트다.
- URL query나 UI selector로 다른 프로젝트를 볼 수 있어야 한다.
- 기본 선택에 실패했을 때만 첫 프로젝트 fallback을 허용한다.
- task/decision/thread/log 조회는 선택된 프로젝트 slug 기준으로 다시 로드한다.

## 5. 구현 메모

현재 `src/core/project-resolver.ts`는 이 원칙의 핵심을 이미 구현 중이다.
남은 일은:
- 테스트 보강
- registry 항목이 많아질 때 우선순위 검토
- 더 긴 경로가 더 우선 매칭되도록 정렬 정책 검토
- dashboard와 web entrypoint에서 현재 path 기반 기본 선택을 재사용할 연결 지점 설계
